# 功能規格：Semantic Model Interpreter — 前端程式執行引擎

**功能分支**: `007-semantic-interpreter`
**建立日期**: 2026-03-05
**狀態**: 草稿
**輸入**: 使用者描述：「基於現有的 SemanticModel / SemanticNode 中間表示，建立一個純前端的程式執行引擎，讓使用者可以直接在瀏覽器中執行積木對應的程式碼。」

## 釐清紀錄

### Session 2026-03-05

- Q: 執行結果面板在 UI 中的位置？ → A: 程式碼面板下方（類似 IDE 終端機）
- Q: 輸入佇列為空時的輸入方式？ → A: console 面板內行內輸入（游標在底部，按 Enter 送出）
- Q: 執行期錯誤訊息是否需要 i18n？ → A: 納入現有 i18n 系統（跟隨 UI 語系切換）

## 使用者情境與測試 *(必要)*

### 使用者故事 1 - 執行程式並查看輸出 (優先級: P1)

身為程式初學者，我希望按下「執行」按鈕後，能在 console 面板立即看到我用積木拼出的程式執行結果，這樣我就能在不離開瀏覽器的情況下驗證我的邏輯。

**為何此優先級**: 核心價值主張是「拼積木，看結果」。沒有基本的執行和輸出功能，其他功能（除錯、變數監看）都沒有意義。

**獨立測試**: 可以用 print 和數字積木建立簡單程式，按下執行，驗證輸出是否出現在 console 面板中。

**驗收情境**:

1. **假設** 程式包含 `print("Hello")`，**當** 使用者按下執行，**則** "Hello" 出現在 console 面板
2. **假設** 程式包含 `int x = 5; print(x);`，**當** 使用者按下執行，**則** "5" 出現在 console 面板
3. **假設** 程式包含算術運算 `print(3 + 4 * 2)`，**當** 使用者按下執行，**則** "11" 出現在 console 面板
4. **假設** 程式包含 count_loop 印出 1 到 5，**當** 使用者按下執行，**則** "1 2 3 4 5" 逐行出現在 console 面板
5. **假設** 程式包含 `if (x > 0) print("positive")` 且 x=3，**當** 使用者按下執行，**則** "positive" 出現在 console 面板
6. **假設** 程式包含遞迴函式（如階乘），**當** 使用者按下執行，**則** 正確結果出現
7. **假設** 程式觸發無窮迴圈，**當** 步數超過最大限制，**則** 執行中止並顯示錯誤訊息

---

### 使用者故事 2 - 虛擬標準輸入 (優先級: P2)

身為學習者，我希望程式能讀取輸入值，這樣我就能測試需要使用者互動的程式（如讀取一個數字並印出它的兩倍）。

**為何此優先級**: 許多初學者練習題需要讀取輸入。沒有此功能，使用者只能執行寫死數值的程式，限制了學習體驗。

**獨立測試**: 可以用 input 積木建立讀取一個值再印出的程式，驗證輸入提示出現且輸出正確。

**驗收情境**:

1. **假設** 程式包含 `input(x); print(x);`，**當** 使用者按下執行，**則** 出現輸入提示詢問 x 的值
2. **假設** 程式需要兩筆輸入，**當** 使用者已在輸入佇列中預填兩個值，**則** 程式直接讀取兩個值且輸出正確
3. **假設** 程式需要輸入但輸入佇列為空，**當** 執行到 input 語句，**則** console 面板底部出現行內輸入游標，使用者直接輸入值並按 Enter 送出
4. **假設** 程式 `input(x)` 中 x 為 int 型別但使用者輸入 "hello"，**則** 出現友善的錯誤訊息提示型別不符

---

### 使用者故事 3 - 逐步執行 (優先級: P3)

身為學習程式概念的學生，我希望能一次執行一個語句地逐步走過程式，看到目前正在執行的積木和程式碼行，這樣我就能理解程式的執行流程。

**為何此優先級**: 逐步執行是最強大的學習工具，但它建立在 US1 和 US2 的基本執行引擎之上。它把工具從「執行看結果」提升到「理解它怎麼運作」。

**獨立測試**: 可以建立一個迴圈程式，反覆按 Step，驗證活動積木和程式碼行的高亮都正確移動。

**驗收情境**:

1. **假設** 程式有 5 個連續語句，**當** 使用者按 Step 5 次，**則** 每次執行一個語句，對應的積木被高亮
2. **假設** 程式正在逐步執行，**當** 一個語句執行時，**則** 程式碼編輯器中對應的行也被高亮
3. **假設** 逐步執行已暫停，**當** 使用者按下繼續，**則** 剩餘程式以選定速度執行到完成
4. **假設** 逐步執行進行中，**當** 使用者按下停止，**則** 執行立即中止，所有高亮清除
5. **假設** 逐步執行中，**當** 使用者將速度調整為「快」，**則** 自動步進間隔縮短，執行更快

---

### 使用者故事 4 - 變數監看面板 (優先級: P4)

身為學習除錯的人，我希望在程式執行時即時看到所有變數的名稱、型別和值的變化，這樣我就能理解資料在執行過程中的變化。

**為何此優先級**: 變數監看補充逐步執行（US3），提供更深入的程式狀態洞察。與逐步執行結合時最有用，但在一般執行時也有價值。

**獨立測試**: 可以執行一個宣告和修改變數的程式，驗證監看面板顯示名稱、型別和值，且值變化時有可見的視覺提示。

**驗收情境**:

1. **假設** 程式包含 `int x = 10;`，**當** 執行過此語句，**則** 監看面板顯示 x、型別 int、值 10
2. **假設** 變數 x 從 5 變為 10，**當** 賦值語句執行，**則** 監看面板的值更新且有短暫的視覺提示（閃爍或變色）
3. **假設** 函式呼叫建立了新的區域範圍，**當** 在函式內部，**則** 監看面板顯示區域變數並標示範圍層級
4. **假設** 函式回傳，**當** 回到呼叫者的範圍，**則** 區域變數從監看面板移除

---

### 邊界情況

- 使用者執行空程式（無語句）時怎麼辦？console 顯示「程式執行完畢」且無輸出。
- 變數在宣告前就被引用時怎麼辦？顯示友善錯誤訊息：「變數 'x' 尚未宣告」，並高亮相關積木。
- 除以零時怎麼辦？顯示友善錯誤：「除以零」，並高亮相關積木。
- 陣列存取超出範圍時怎麼辦？顯示友善錯誤：「陣列索引超出範圍」，並高亮相關積木。
- 執行中使用者修改積木時怎麼辦？自動停止執行並顯示訊息：「程式已修改，請重新執行。」
- 函式無限遞迴呼叫自己時怎麼辦？步數限制會觸發，顯示：「已超過最大執行步數（可能是無窮迴圈或過深的遞迴）。」
- break 或 continue 在迴圈外使用時怎麼辦？顯示友善錯誤：「break/continue 只能在迴圈內使用。」
- print 與 endl 一起呼叫時怎麼辦？在 console 輸出中附加換行。
- 巢狀函式呼叫如 `print(f(g(x)))` 怎麼辦？每個呼叫依正確順序解析（最內層先執行）。

## 需求 *(必要)*

### 功能需求

- **FR-001**: 系統必須解釋所有 universal SemanticNode 概念：program、var_declare、var_assign、var_ref、number_literal、string_literal、arithmetic、compare、logic、logic_not、if（含可選的 else_body）、count_loop、while_loop、break、continue、func_def、func_call、return、print、input、endl、array_declare、array_access
- **FR-002**: 系統必須支援基本型別語義：int（整數運算帶截斷）、float/double（浮點運算）、char、string（串接）、bool
- **FR-003**: 系統必須實作作用域（scope）：函式建立新的 scope；迴圈和 if 區塊繼承父 scope
- **FR-004**: 系統必須支援遞迴函式呼叫，每次呼叫有獨立的 scope
- **FR-005**: 系統必須強制執行可設定的最大步數限制（預設：10,000 步），超過時中止執行並顯示明確訊息
- **FR-006**: 系統必須在程式碼面板下方的虛擬 console 面板中顯示 print 輸出（類似 IDE 終端機佈局，不改動現有左右雙欄結構）
- **FR-007**: 系統必須支援透過預填輸入佇列或佇列為空時在 console 面板內的行內輸入來進行 input（游標出現在 console 底部，使用者輸入後按 Enter 送出）
- **FR-008**: 系統必須支援逐步執行，每按一次 Step 剛好前進一個語句
- **FR-009**: 系統必須在逐步執行期間，在 Blockly 工作區中高亮目前正在執行的積木
- **FR-010**: 系統必須在逐步執行期間，在程式碼編輯器中高亮對應的程式碼行
- **FR-011**: 系統必須在工具列提供「執行」、「逐步」、「停止」和「暫停/繼續」控制按鈕
- **FR-012**: 系統必須顯示變數監看面板，列出所有可見變數的名稱、型別和目前值
- **FR-013**: 系統必須在變數值變化時提供視覺指示器（閃爍或變色）
- **FR-014**: 系統必須顯示目前執行狀態：執行中、已暫停、已完成、或錯誤
- **FR-015**: 系統必須為執行階段錯誤（未宣告變數、除以零、型別不符、索引超出範圍）顯示使用者友善的錯誤訊息，並納入現有 i18n 系統（錯誤訊息跟隨 UI 語系切換）
- **FR-016**: 系統必須支援可調整的自動步進速度，至少 3 個等級（慢、中、快）
- **FR-017**: 系統必須完全在瀏覽器中運作，不需任何後端資源
- **FR-018**: 系統必須直接解釋 SemanticModel，而非透過產生的文字程式碼
- **FR-019**: 系統必須妥善處理語言特有概念（如 cpp:include、cpp:using_namespace），在解釋時跳過它們（它們沒有執行期效果）

### 關鍵實體

- **ExecutionContext（執行環境）**: 包含變數作用域、呼叫堆疊、步數計數器和執行狀態的執行期狀態
- **Scope（作用域）**: 變數名稱到其帶型別值的對映，帶有對父作用域的參考
- **RuntimeValue（執行期值）**: 執行期間使用的帶型別值（int、float、double、char、string、bool、void）
- **IOSystem（輸入輸出系統）**: 虛擬 stdin 佇列和 stdout 緩衝區，用於程式 I/O
- **StepController（步進控制器）**: 管理執行模式（執行/逐步/暫停）、速度和逐步推進

## 成功標準 *(必要)*

### 可衡量成果

- **SC-001**: 使用者可以在按下執行後 1 秒內看到基本程式（變數 + print）的輸出
- **SC-002**: 所有 22 個 universal SemanticNode 概念都被正確解釋（由涵蓋每個概念的測試套件驗證）
- **SC-003**: 逐步執行每按一次正確前進一個語句，積木和程式碼高亮在 200ms 內更新
- **SC-004**: 變數監看面板在每步之後 200ms 內更新，顯示所有可見變數的正確值
- **SC-005**: 當程式超過最大步數時，無窮迴圈保護在 2 秒內觸發
- **SC-006**: 最多 5 層遞迴函式呼叫的程式能正確執行
- **SC-007**: 執行期錯誤訊息以使用者友善的語言顯示，而非技術術語
- **SC-008**: 直譯器完全在瀏覽器中執行，執行期間零網路請求

## 假設

- 直譯器操作的是由現有的 Blockly-to-SemanticModel 管線（透過 CppLanguageAdapter）產生的 SemanticModel，不需要新的解析。
- if 概念透過 else_body 子節點的有無來處理 if 和 if-else 兩種情況（符合現有語義模型設計）。
- 語言特有概念（如 cpp:include、cpp:using_namespace）在執行期是空操作（no-op），因為它們是編譯指令。
- func_def 概念將參數資訊儲存在 params 屬性中，以逗號分隔的字串格式（符合現有 adapter 輸出）。
- 整數除法向零截斷（C/C++ 語義）。
- 輸入佇列透過 UI 中的文字區域預填；佇列用完時在 console 面板內顯示行內輸入游標。
- 10,000 步的效能目標對初學者程式已足夠；進階使用者可透過 UI 增加限制。
